# Fix Tasks Not Appearing in Active Tasks or Task History

## What
When dispatching tasks to multiple projects, the tasks execute (based on Running/Pending counts in header) but do not appear in the Active Tasks pane or Task History.

## Screenshot 1 Description (During execution)
- PROJECTS pane: 4 projects selected (Tiny Music Library, Vintage Radio have orange borders visible)
- DISPATCH TASK pane: Shows "4 projects selected"
- Header: Connected, Running: 1, Pending: 2 (indicates tasks ARE running)
- ACTIVE TASKS pane: **EMPTY** (should show running tasks)
- TASK HISTORY pane: Visible at bottom, appears empty

## Screenshot 2 Description (After completion)
- Same project selection (Tiny Music Library, Vintage Radio selected)
- DISPATCH TASK pane: Still shows "4 projects selected"
- Header: Running: 0, Pending: 0 (tasks have completed)
- ACTIVE TASKS pane: Shows "No active tasks. Select a project and dispatch a task."
- TASK HISTORY pane: **EMPTY** (should show completed tasks)

## Observed Issues
1. Tasks ARE executing (Running/Pending counts change in header)
2. Active Tasks pane never shows the running tasks
3. Task History is empty after tasks complete
4. Disconnect between backend task tracking and frontend UI display

## Expected Behavior
- Active Tasks should show all running/pending tasks with their status
- Task History should populate with completed tasks
- UI should reflect the same state as the Running/Pending counts

## Context
This appears to be a different issue from REQ-004. In REQ-004, only one task was created. Here, tasks seem to be created and running (counts update) but the UI components (Active Tasks, Task History) are not receiving or displaying the task data.

Possible causes:
- WebSocket events not being sent for task creation/updates
- Frontend not subscribing to correct events
- Task data structure mismatch between backend and frontend

## Triage

**Route: B** - Medium

**Reasoning:** Alpine.js reactivity issue - array mutations weren't triggering DOM updates. Need to ensure all array modifications create new array references.

## Exploration

### Root Cause

Alpine.js tracks reactivity through proxy references. When arrays are mutated in-place:
- `array[index] = newValue` - doesn't change array reference
- `array.unshift(item)` - mutates in place
- `object.property = value` - doesn't trigger re-render

These patterns don't trigger Alpine.js re-renders even though the data changed.

### Fix Strategy

Convert all in-place mutations to spread operations that create new array references:
- `updateTask()` - use `map()` instead of index assignment
- `task:output` - use `map()` instead of direct object mutation
- `taskHistory` - use spread instead of `unshift()`

*Generated by exploration*

## Implementation Summary

Modified `public/js/app.js`:

**Fixed `updateTask()` method:**
- Changed from `this.activeTasks[index] = {...}` (in-place)
- To `this.activeTasks = this.activeTasks.map(...)` (new array)

**Fixed `task:output` handler:**
- Changed from direct object mutation `task.output = ...`
- To `this.activeTasks = this.activeTasks.map(...)` with spread

**Fixed `taskHistory` update:**
- Changed from `this.taskHistory.unshift(data.task)`
- To `this.taskHistory = [data.task, ...this.taskHistory].slice(0, 100)`

All array modifications now create new references, ensuring Alpine.js detects changes and re-renders the DOM.
