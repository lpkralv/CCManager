# Cross-User Sandbox Execution

## What
Explore whether CCManager can run from the `louisslothouber` (admin) user account but dispatch and control Claude Code tasks that execute under the `agenthome` (sandbox) user account. The goal is to manage sandbox projects without constantly switching macOS users. This is an investigation only. Do not modify the app yet.

## Context
The user has two macOS accounts:
- **louisslothouber**: Main admin account, signed in to Apple ID, used for daily work
- **agenthome**: Restricted sandbox account with access to the `sandbox` directory only, not signed in to any Apple ID, used for running Claude Code safely with bypassed permissions

Currently, managing Claude Code projects requires switching to the agenthome user. The user wants to run CCManager from their main account and have it execute Claude Code processes in the sandbox user's context.

This needs research into: `sudo -u agenthome`, `launchd` per-user agents, SSH to localhost as agenthome, file permission implications on the shared sandbox directory, whether Claude Code CLI respects the executing user's environment/config, and any security considerations.

## Implementation Summary

### Analysis: Cross-User Sandbox Execution

The goal: run CCManager from `louisslothouber` (admin account) but have Claude Code tasks execute as `agenthome` (sandbox account).

**Note:** The request asks about running from louisslothouber → executing as agenthome. Currently the app runs as agenthome. The direction matters because the approaches differ.

---

### Current Architecture

The process spawner calls `spawn("claude", ...)` which inherits the current user. There's no mechanism to specify a different execution user. Claude Code's CLI, config, and API keys are all user-specific (stored in `~/.claude/`).

### Approaches

#### Approach 1: SSH to localhost (Recommended)

Run CCManager on `louisslothouber`. Modify the process spawner to use `ssh agenthome@localhost claude -p "prompt" ...` instead of direct `spawn("claude", ...)`.

**Setup required:**
1. Enable macOS Remote Login (System Settings → General → Sharing → Remote Login)
2. Generate SSH key on louisslothouber: `ssh-keygen`
3. Copy to agenthome: `ssh-copy-id agenthome@localhost`
4. Test: `ssh agenthome@localhost claude --version`

**Pros:**
- Clean separation — processes genuinely run as agenthome with agenthome's Claude config/API keys
- No password prompts after key setup
- Standard, well-understood mechanism
- CCManager runs natively on louisslothouber's desktop

**Cons:**
- Requires enabling SSH (minor security surface)
- stdout/stderr streaming needs SSH session management
- One-time setup required

**Code change:** Modify `process-spawner.ts` to optionally wrap commands in `ssh user@localhost`. Add `executionUser` to project/task model.

#### Approach 2: Passwordless sudo

Add a sudoers rule: `louisslothouber ALL=(agenthome) NOPASSWD: /Users/agenthome/.local/bin/claude`

Then spawn as: `sudo -u agenthome -i claude -p "prompt" ...`

**Pros:** Simple, no SSH daemon needed.
**Cons:** Requires editing `/etc/sudoers` (risky if misconfigured). The `-i` flag is needed to load agenthome's environment/PATH, which adds complexity.

#### Approach 3: Run CCManager Server as agenthome, UI as louisslothouber

Keep the server running under agenthome (via launchd agent or the Electron app in agenthome's session). From louisslothouber, access it via browser at `http://localhost:3000`.

**Pros:** Zero code changes. Works today. The server already binds to all interfaces.
**Cons:** Requires agenthome to be logged in (Fast User Switching keeps both sessions active). You'd need to start the server in agenthome's session once.

#### Approach 4: launchd Service as agenthome

Create a launchd plist that runs the CCManager server as agenthome at boot. Access from any user via browser.

**Pros:** Always available, no user switching needed.
**Cons:** Requires root to install a system-level LaunchDaemon (or agenthome login for a LaunchAgent). Server runs even when not needed.

---

### Recommendation

**Approach 3 is the path of least resistance** — it works today with zero code changes. Just:
1. Log into agenthome (Fast User Switching keeps both sessions alive)
2. Start the server (`npm run dev` or launch CCManager.app)
3. Switch back to louisslothouber
4. Open `http://localhost:3000` in your browser

The server stays running in agenthome's background session. Tasks execute as agenthome. You manage everything from louisslothouber's browser.

**If you want a more permanent solution**, Approach 1 (SSH) is the cleanest for true cross-user execution from louisslothouber's native environment, but requires code changes to the process spawner and one-time SSH setup.
