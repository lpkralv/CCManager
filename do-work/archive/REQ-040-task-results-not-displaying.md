# Task Results Not Displaying in Log Area

## What
Fix bug where task results stop appearing in the log/results area after a failed task. New tasks (both successful and failed) do not show any results in the log area.

## Context
- Initial failed task occurred when Claude Code was not logged in on the agent account
- That failed task created a "failed" entry in the log/results area
- After that first failure, all subsequent tasks do not display results in the log/results area
- This happens regardless of whether new tasks succeed or fail
- Server is still running (in agent account)
- Web app is still accessible at http://localhost:3000 (from main account)
- The failure happens very quickly (within ~1 second), suggesting it's not reaching Claude Code
- User tried multiple times with same result

## Assets
None

## Triage

**Route: B** - Medium

**Reasoning:** Clear bug report with good reproduction steps, but need to explore task dispatch/results handling in both server and frontend to find where state gets corrupted after a failed task.

## Exploration

Key findings:

1. **WebSocket event forwarding has no error handling** (`src/server/websocket/index.ts:29-37`): `client.send()` calls in the event listener have no try-catch. If a send fails, the event listener callback crashes silently, breaking event distribution for all clients.

2. **Missing `task:retrying` handler in frontend** (`public/js/app.js:213-262`): Server emits `task:retrying` events but the frontend switch statement has no case for it.

3. **No error handling on initial WebSocket state send** (`src/server/websocket/index.ts:13-18`): The initial state send on connection has no error handling.

4. **Critical bug flow**: A failed task could cause `client.send()` to throw → event listener crashes → all subsequent events stop reaching frontend → log area stays empty.

*Generated by Explore agent*

## Implementation Summary

- **`src/server/websocket/index.ts`**: Added try-catch around all `client.send()` calls. Initial state send on connection is now wrapped in try-catch. Event forwarding loop catches per-client send errors, logs them, and removes dead clients from `wss.clients` to prevent repeated failures.
- **`public/js/app.js`**: Added missing `task:retrying` case to the WebSocket message handler switch statement, calling `updateTaskInPlace(data.task)` to update task status in the UI.
- **`src/server/websocket/index.test.ts`**: Added 5 regression tests covering: initial send failure resilience, handler registration after send failure, event forwarding continues when one client fails, dead client cleanup, and subsequent event delivery after cleanup.

**Root cause**: Unguarded `client.send()` in the WebSocket event forwarding loop could throw an uncaught exception, crashing the entire event listener callback and preventing all subsequent task events from reaching any frontend client.
