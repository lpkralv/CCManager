# Fix Active Tasks Pane Only Displaying First Task

## What
When dispatching tasks to multiple projects, the ACTIVE TASKS pane is sized to accommodate all tasks (suggesting it knows about them), but only the first task is ever displayed.

## Screenshot Description
- PROJECTS pane: Multiple projects selected (visible: Pocketwatch ESP32 Software, Pocketwatch Software, Ship2Ship, Tiny Music Library, Vintage Radio - all with orange selection borders)
- DISPATCH TASK pane: Shows "16 projects selected"
- Header: Running: 6, Pending: 16 (confirms 22 total tasks are tracked)
- ACTIVE TASKS pane:
  - Pane appears to be sized larger (more vertical space than usual)
  - Only ONE task displayed: "C64 Next" with PENDING status
  - Prompt shown: "If a do-work directory does not exist in your project directory, run the /do-work skill."
  - Time: 9:24:24 PM
  - Large empty space below the single task card

## Observed Issues
1. Header counts show 6 running + 16 pending = 22 tasks being tracked
2. Active Tasks pane only renders 1 task card
3. Pane sizing suggests it may be reserving space for more tasks
4. The 21 other tasks are not visible

## Expected Behavior
- All 22 active tasks should be displayed in the ACTIVE TASKS pane
- Each task should have its own card showing project name, status, prompt, and time
- Pane should be scrollable if tasks exceed visible area

## Context
This is related to REQ-004 (only first project dispatched) and REQ-006 (tasks not showing in UI). The backend appears to be tracking all tasks correctly (based on Running/Pending counts), but the frontend rendering only shows one task.

Possible causes:
- React state only holding first task
- Array being overwritten instead of appended
- Rendering logic only processing first element
- WebSocket updates replacing instead of merging

### Root Cause Found
In `public/js/app.js` (lines 91-96), the WebSocket `initial` message handler unconditionally overwrites the `activeTasks` array:

```javascript
if (data.type === 'initial') {
  console.log(`Initial state: ${data.tasks?.length || 0} active tasks`);
  this.activeTasks = data.tasks || [];  // <- OVERWRITES entire array
  this.updateCounts();
  return;
}
```

The bug manifests through a race condition:
1. Client connects → Server sends `initial` message with `getActiveTasks()` snapshot
2. Client rapidly dispatches multiple tasks → All tasks are queued quickly
3. WebSocket reconnection or page reload occurs during task creation
4. New `initial` message arrives with only the tasks at that exact moment
5. Line 93 overwrites the entire `activeTasks` array with this partial snapshot
6. Result: Frontend shows only 1 task despite backend tracking 22

### Related Fixes from REQ-004/006
Previous fixes properly addressed other handlers:
- ✅ `task:created` uses spread operator (line 102)
- ✅ `updateTask()` uses map() (lines 143-149)
- ✅ Task output updates use map() (lines 112-117)
- ✅ Task history uses spread (line 132)

But the `initial` handler was not fixed.

### Fix Required
The `initial` message handler needs to merge tasks instead of replacing them.

## Implementation Summary

**File Modified:** `public/js/app.js`

**Changes Made:**
Modified the WebSocket `initial` message handler (lines 91-111) to merge tasks instead of replacing them:

1. If `activeTasks` is empty, directly assign incoming tasks (same behavior as before)
2. If `activeTasks` already has tasks, merge by adding only tasks that aren't already present (using Set of existing IDs)
3. Use spread operator (`[...this.activeTasks, ...newTasks]`) for Alpine.js reactivity
4. Added improved logging to help debug future issues
