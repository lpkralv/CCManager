# Fix Task Counts Not Updating After Completion

## What
After sending a prompt to a project, the running and pending counts update correctly during execution. However, when the task completes, the counts fail to update - it continues showing "pending: 0, running: 1" instead of resetting to reflect the completed state.

## Context
- Running/pending counts update appropriately when task starts
- After task completion, counts are stale (still shows running: 1)
- Likely a WebSocket event or state update issue

## Requirements
1. Investigate the task completion flow and count update mechanism
2. Plan the fix to ensure counts update properly on task completion
3. Verify the plan addresses the issue
4. Implement the fix

## Exploration

### Root Cause

In `public/js/app.js`, the WebSocket message handler:
1. Receives `task:completed` event
2. Calls `updateTask()` which sets status to "completed" but keeps task in `activeTasks`
3. Immediately calls `updateCounts()`
4. `updateCounts()` filters for `status === 'running'` and `status === 'pending'`
5. The completed task doesn't match either filter, but it's still in the array
6. Task is only removed from `activeTasks` 3 seconds later

### Key Files

- `public/js/app.js:112-127` - WebSocket handler for completion events
- `public/js/app.js:139-142` - `updateCounts()` method filters only running/pending
- Backend is working correctly - events are sent properly

### Fix Strategy

Call `updateCounts()` AFTER removing the task from `activeTasks`, not before.

## Implementation Summary

Modified `public/js/app.js`:

**Change:** Added `this.updateCounts()` call inside the setTimeout block for task completion/failed/cancelled events (line 122).

The fix ensures `updateCounts()` is called both:
1. Immediately when the task status changes (line 127)
2. After the 3-second delay when the task is removed from `activeTasks` (new line 122)

This guarantees counts are refreshed when the task is moved to history.
