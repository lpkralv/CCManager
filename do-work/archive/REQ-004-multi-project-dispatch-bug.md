# Fix Multi-Project Dispatch - Only Dispatches to First Project

## What
When multiple projects are selected and a task is dispatched, only the first project (ClaudeCodeManager) receives the task. The remaining 9 selected projects do not get the task dispatched. Additionally, task history is not updated.

## Screenshot Description
- PROJECTS pane shows multiple projects with orange/highlighted borders indicating selection (Pocketwatch ESP32 Software, Pocketwatch Software, Tiny Music Library, Vintage Radio are visibly selected)
- DISPATCH TASK pane shows "10 projects selected" in the Projects field
- ACTIVE TASKS pane only shows ONE task: ClaudeCodeManager with PENDING status and prompt "commit and push if needed." at 5:26:44 PM
- Header shows "Running: 4, Pending: 13" which seems inconsistent with only 1 visible active task

## Observed Issues
1. Only first project (ClaudeCodeManager) received the dispatched task
2. Other 9 selected projects did not receive tasks
3. Task history was not updated
4. Running/Pending counts in header may be inaccurate

## Expected Behavior
- All 10 selected projects should have received independent tasks
- 10 task cards should appear in ACTIVE TASKS
- Task history should be updated for all dispatched tasks

## Context
This is a bug in the multi-project dispatch feature (related to REQ-003). The dispatch logic likely only processes the first project in the selection array.

## Exploration

### Code Review Findings

**Frontend dispatch (`public/js/app.js:192-235`):**
- Code correctly maps over `selectedProjects` and creates parallel fetch calls
- Uses `Promise.allSettled()` to wait for all results
- Error handling looks correct

**Backend task creation (`src/server/routes/tasks.ts`):**
- Each POST creates task via `taskManager.createTask()`
- No obvious issues with concurrent requests

**Task Manager (`src/services/task-manager.ts`):**
- `createTask()` pushes to pendingTasks and emits `task:created`
- `processQueue()` starts up to 3 concurrent tasks

**WebSocket (`src/server/websocket/index.ts`):**
- Broadcasts all task events to connected clients

### Analysis

The code path appears correct. The issue may be:
1. **Project ID validation failing** for 9 projects (silently via error handling)
2. **WebSocket message delivery issue**
3. **Alpine.js reactivity not updating** the DOM

Most likely: Need to verify that all 10 POST requests are actually being sent and that the backend receives them. The current error handling catches errors per-project but may not surface them clearly.

### Fix Strategy

Add better debugging/logging to identify where the failure occurs. If frontend is sending all requests but backend fails, improve error messages. If WebSocket is the issue, investigate connection state.
